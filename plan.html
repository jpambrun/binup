Bun GitHub Binary Installer

JSON config/spec shape

The editable config contains packages only. Runtime defaults such as ~/bin, gh auth token, GitHub API URL, concurrency, platform token mapping, archive preferences, and file exclusions are hardcoded in the Bun script.

{
  "packages": [
    {
      "repo": "sharkdp/fd",
      "version": "v10.4.2",
      "binaries": [{ "name": "fd" }]
    },
    {
      "repo": "BurntSushi/ripgrep",
      "version": "latest",
      "binaries": [{ "name": "rg" }]
    }
  ]
}

Installer workflow

PhaseWhat happensParallelism
1. Load configRead package-only JSON, then apply hardcoded defaults for install directory, platform support, GitHub settings, and asset exclusions.No
2. AuthRun gh auth token; keep token in memory only; send as GitHub API bearer token.No
3. Check releasesFetch latest or pinned GitHub release metadata for every repo.Yes, bounded by hardcoded check concurrency
4. Select assetScore release assets for requested OS/arch, archive type, and package hints.Yes
5. Skip currentCompare selected release tag + asset id/name/size with local state. If unchanged and binaries exist, skip download.Yes
6. Download outdatedDownload only changed assets to a temp directory, extract, find binaries, chmod, then atomically replace.Yes, bounded by hardcoded download concurrency
7. Save stateWrite ~/.local/state/my-installer/installed.json after successful installs.No

GitHub asset matching

The resolver will normalize platform tokens and score assets instead of requiring exact filenames.

OS tokens
linuxunknown-linuxdarwinmacosapple-darwin
x64 tokens
x86_64amd64x64
arm64 tokens
aarch64arm64
Preferred assets
tar.gztgztar.xzzipraw executable

Checksum, signature, SBOM, package-manager formats, and source archives are excluded by hardcoded defaults unless the script is changed.

Finding the binary inside an asset

  1. If package config lists binaries[].path, use it.
  2. Otherwise extract/list the archive and search executable files whose basename matches binaries[].name.
  3. If exactly one executable exists, use it as a fallback.
  4. If ambiguous, fail with a useful message and suggest a config override.

Offline artefact-selection unit tests

Unit tests must not call GitHub. Before implementation, run a one-time fixture collection step against the current package list and capture representative release metadata for each package: release tag, asset id, asset name, size, content type, and browser download URL. The captured examples will live inside installer.spec.ts as test fixtures, not in the editable JSON config.

The asset resolver will be tested against those fixtures for all supported target platforms: linux-x64, linux-arm64, darwin-x64, and darwin-arm64. Tests should cover successful matches, missing-platform failures, binary rename cases, explicit select hints, duplicate assets, and excluded checksum/signature/SBOM files.

Download-only-when-new logic

state key = repo + binary + os + arch
installed record = {
  releaseTag,
  assetId,
  assetName,
  assetSize,
  sha256?,
  installedPath,
  installedAt
}

A package is skipped only when the selected release/asset matches state and all expected binaries are still present. The first implementation can rely on GitHub asset metadata; optional checksum support can be added later when upstream publishes checksums.

Implementation milestones

  1. Scaffold: create exactly two source files: installer.ts for implementation and installer.spec.ts for tests.
  2. Config loader: validate package-only JSON and platform normalization.
  3. GitHub client: release lookup with gh auth token and retry/rate-limit messages.
  4. Fixture collection: gather current GitHub release asset examples for every package once and embed them in installer.spec.ts so unit tests are offline.
  5. Resolver: asset scoring for linux/darwin × x64/arm64 and per-package select hints, covered by fixture-based tests.
  6. State/skip: implement local installed-state comparison.
  7. Installer: parallel downloads, archive extraction, binary discovery, atomic replacement.
  8. Tests: run bun test installer.spec.ts; tests use embedded release fixtures and never hit GitHub.

Review decisions

Recommended defaults:

  • Use pinned aqua versions where available; use latest for install.sh-only packages.
  • Install into hardcoded ~/bin to preserve current behavior.
  • Do not store the GitHub token; execute hardcoded gh auth token at runtime.
  • Use p-map for bounded parallel release checks/downloads; avoid p-queue unless later dynamic queueing/rate-limit control is needed.
  • Make unresolved package assets fail the run by default, with --best-effort as an optional later flag.

Known follow-ups

  • Some binaries need verification or overrides because repo names differ from binary names, e.g. ClementTsang/bottom → btm, Wilfred/difftastic → difft, alexpasmantier/television → tv.
  • Some upstreams publish multiple binaries from one repo, e.g. weave and atlassian-cli; the spec keeps these as separate install entries.
  • Non-GitHub package sources are out of scope for the first pass; the current inventory appears GitHub-release based.
